# CLAUDE.md

## コミット規約

コミットメッセージは [Conventional Commits](https://www.conventionalcommits.org/ja/) に従う。

```
<type>(<scope>): <subject>

<body>

<footer>
```

### type 一覧

| type       | 用途                                                   |
| ---------- | ------------------------------------------------------ |
| `feat`     | 新機能・新しいツールや実験の追加                       |
| `fix`      | バグ修正                                               |
| `docs`     | ドキュメントのみの変更                                 |
| `style`    | 動作に影響しない整形（フォーマット・空白など）         |
| `refactor` | 挙動を変えないコード改善                               |
| `perf`     | パフォーマンス改善                                     |
| `test`     | テストの追加・修正                                     |
| `chore`    | ビルド・依存・設定など雑多な変更                       |
| `exp`      | 実験的な検証コード・試行（結果が残らない可能性がある） |

### subject（要約行）

- 日本語で簡潔に。「何をしたか」を動詞で書く。
- 末尾に句点（。）は付けない。50 文字程度を目安にする。

### body / footer（任意）

- **なぜ** その変更をしたか、背景や判断理由を書く。
- 破壊的変更は `BREAKING CHANGE:` を footer に記載する。
- 関連 Issue があれば `Refs #123` / `Closes #123` を footer に書く。

## コミットの粒度の判断基準

**1 コミット = 1 つの論理的にまとまった変更** を原則とする。

### 分けるべきケース（別コミットにする）

- type が異なる変更（例: `feat` と `docs`、`fix` と `refactor` を混在させない）。
- 対象サブプロジェクトが異なる変更。
- 「機能追加」と「既存コードのリファクタリング」を同時にやった場合は分割する。
- フォーマット・リネームのような大量の機械的差分は、ロジック変更と分けて単独コミットにする。

### まとめてよいケース（1 コミットにする）

- 1 つの変更を成立させるために不可欠な複数ファイルの修正（実装＋それに対応するテストなど）。
- そのコミット単体で **テストが壊れない** 範囲。

### 目安

- 「このコミットの subject を `and` / 「〜と〜」でつながないと説明できない」場合は、分割を検討する。
- WIP（作業途中）の状態をそのまま積まない。区切りごとに、それ自体で意味の通る単位でコミットする。
- 1 コミットが巨大になりすぎたら、レビューしやすい単位に分割できないか見直す。
